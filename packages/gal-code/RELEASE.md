# GAL Code Release Checklist

This document provides a step-by-step guide for releasing GAL Code v0.1.1 and subsequent versions.

## 1. Pre-release Checklist

### Repository State

- [ ] All PRs for the release are merged to `main`
- [ ] CI checks pass on `main` or the release PR (test.yml, typecheck.yml)
- [ ] No blocking issues remain in the milestone
- [ ] CHANGELOG.md is updated (auto-generated by `script/version.ts`)

### Package Configuration

- [ ] Package name is `@gal-run/code`
- [ ] `packages/gal-code/package.json` is publishable
- [ ] Version in `packages/gal-code/package.json` is correct
- [ ] npm publishing rights confirmed for the package scope

### Secrets & Credentials

- [ ] `NPM_TOKEN` secret is set in repository secrets
- [ ] `GAL_CODE_APP_ID` and `GAL_CODE_APP_SECRET` are configured
- [ ] Azure Trusted Signing secrets are configured (for Windows signing)
- [ ] Apple certificates are configured (for macOS signing)
- [ ] `AUR_KEY` is configured (for Arch Linux AUR publishing)

### CI/CD Verification

- [ ] Test workflow runs green on PR
- [ ] Typecheck workflow runs green
- [ ] Build workflow completes successfully on `dev` branch

## 2. Version Bump Process

### Automated (Recommended)

Trigger the publish workflow with version bump input:

```bash
# Via GitHub UI: Actions > publish > Run workflow
# Or via CLI:
gh workflow run publish.yml --repo gal-run/gal-code --ref main -f bump=patch
```

Options:

- `major` - Breaking changes (1.0.0 → 2.0.0)
- `minor` - New features (1.0.0 → 1.1.0)
- `patch` - Bug fixes (1.0.0 → 1.0.1)

### Manual Version Override

To specify an exact version:

```bash
gh workflow run publish.yml --repo gal-run/gal-code --ref main -f version=0.1.1
```

### Branch Promotion

Stable releases are mainline-only. The manual workflow must run on `main`, and it now refuses non-`main` refs for stable version bumps so the release commit fast-forwards `main` instead of creating a side commit.

The publish workflow also triggers on:

- Push to `ci` branch → CI testing
- Push to `dev` branch → Development release
- Push to `beta` branch → Beta release
- Push to `snapshot-*` branches → Snapshot builds

## 3. npm Publish Steps

### Automated Publishing

The `publish.yml` workflow handles npm publishing automatically:

1. **Build CLI** (`build-cli` job):
   - Builds platform binaries (darwin, linux, windows)
   - Uploads artifacts for signing

2. **Sign Windows** (`sign-cli-windows` job):
   - Azure Trusted Signing for Windows binaries
   - Repacks signed archives

3. **Publish** (`publish` job):
   - Logs into npm registry
   - Runs `packages/gal-code/script/publish.ts`
   - Publishes to npm with provenance

### Manual Publishing (Fallback)

If CI needs manual intervention:

```bash
# From repo root
cd packages/gal-code

# Build the package
bun run build

# Login to npm
npm login --registry https://registry.npmjs.org

# Publish (dry-run first)
npm publish --dry-run

# Publish for real after creating the generated package in dist
npm publish --access public
```

### Verify npm Package

```bash
# Check package is live
npm view @gal-run/code

# Install globally to test
npm install -g @gal-run/code
gal-code --version
```

## 5. Rollback Procedure

### If npm package is broken:

1. **Deprecate the version**:

   ```bash
   npm deprecate @gal-run/code@0.1.1 "Critical bug - please use 0.1.2"
   ```

2. **Unpublish** (within 72 hours only):

   ```bash
   npm unpublish @gal-run/code@0.1.1
   ```

3. **Release a fix**:
   ```bash
   # Fix the issue, then release a patch
   gh workflow run publish.yml --ref main -f bump=patch
   ```

### If GitHub release is broken:

1. **Delete the release**:

   ```bash
   gh release delete v0.1.0 --repo gal-run/gal-code --yes
   ```

2. **Delete the tag**:

   ```bash
   git push origin --delete v0.1.0
   ```

3. **Re-run the release** after fixes are merged.

### If CI is blocked:

1. **Manual release** (see Section 3)
2. **Create GitHub release manually**:
   ```bash
   gh release create v0.1.0 --title "v0.1.0" --notes-file RELEASE_NOTES.md
   ```
